# AGENTS.md

Цей документ є обов'язковою інструкцією для OpenCode під час роботи з цим репозиторієм.
Його мета — забезпечити стабільні, передбачувані та масштабовані Next.js додатки без технічного боргу.

## Загальні принципи

- Працюй детерміновано: жодних припущень без підтвердження в коді або контрактах.
- Типобезпечність — не опція. TypeScript у строгому режимі, без `any`.
- Один source of truth: контракт, схема або конфігурація завжди важливіші за імплементацію.
- Якщо вимоги або контракт не повні — СТОП і зафіксуй проблему, не імпровізуй.

## Архітектура Next.js (App Router)

- Використовуй App Router (Next.js 13+).
- Server Components — за замовчуванням. Client Components лише коли це необхідно (інтерактивність, браузерні API).
- Дані отримуй на сервері, якщо немає жорсткої потреби в клієнтському фетчінгу.
- Логіка не живе в JSX: бізнес-правила, трансформації та запити — поза компонентами.
- Використовуй стандартні файли маршруту: `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`.
- Не зберігай секрети або ключі в клієнтському коді. ENV змінні з `NEXT_PUBLIC_` — тільки для публічних значень.

## Стан і дані

- Server State (дані з API) ≠ Client State (UI-стан).
- Для server state використовуй запити через API-шар (наприклад, React Query).
- Форми та мутації мають явну валідацію і контроль помилок.
- Ніяких сайд-ефектів у рендерах.

## API інтеграція (ОБОВ'ЯЗКОВО)

- Працюй з бекендом виключно через контракт OpenAPI 3.x.
- Єдиний актуальний контракт знаходиться тут: `.vibe/openapi/openapi.json`.
- Якщо контракт змінено — обов'язково перегенеруй типи/клієнт (`npm run api:generate`).
- Якщо контракт відсутній або неузгоджений — СТОП. Перший крок — узгодити OpenAPI на бекенді.
- Заборонено вигадувати поля, ендпоінти або поведінку API.
- Усі API-запити проходять через єдиний шар (apiClient / services / queryFn).
- Заборонено використовувати `fetch` або `axios` напряму в компонентах.
- Обробляй усі можливі статус-коди явно, не лише happy path:
  - `400` — помилка вхідних даних
  - `401` — неавторизований доступ
  - `403` — заборонено
  - `404` — ресурс не знайдено
  - `422` — помилка бізнес-правил
  - `500` — помилка сервера
- Кожен статус має мапитись на зрозумілу UX-реакцію або повідомлення.

## Обробка помилок і UX

- Помилки API не ігноруються і не логуються "про всяк випадок".
- Повідомлення для користувача мають бути зрозумілими і не технічними.
- `401` / `403` мають чітку стратегію (редирект, logout, блокування дії).

## Якість коду

- Заборонено використовувати `// @ts-ignore` без явного обґрунтування.
- Немає магічних чисел і рядків без пояснення.
- Дотримуйся існуючих патернів репозиторію, не вводь нові без потреби.

## OpenCode НІКОЛИ не повинен

- Обходити або ігнорувати OpenAPI контракт.
- "Латати" проблеми бекенду прихованою логікою на фронтенді.
- Додавати нові API-залежності без чіткої необхідності.
- Маскувати помилки замість їх коректної обробки.
